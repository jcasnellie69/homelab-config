# AGENTS.md

## Purpose

This repository uses **named agent processes** to perform structured, repeatable work using tools like Jules.

Agents are used to:

* maintain repository structure and clarity
* monitor drift and sync issues
* generate reports and summaries
* improve navigation and documentation
* protect against data loss during active evolution

This file defines how agents operate in this repository.

---

# Agent Model

## Naming Standard

All agent processes must include the word `agent`.

Pattern:

```text
<domain>-<role>-agent[-mode]
```

Examples:

* ops-hand-brief-agent-report
* repo-sync-monitor-agent-monitor
* repo-recycle-audit-agent-review
* docs-canvas-agent-maint
* delivery-scrum-reporter-agent-report

---

## Agent Types

### Reporting Agents

* Produce reports and summaries
* Prefer writing to `docs/reports/`
* Avoid creating pull requests unless necessary

Examples:

* ops-hand-brief-agent-report
* delivery-scrum-reporter-agent-report

---

### Maintenance Agents

* Perform small, safe improvements
* Keep navigation, indexes, and structure aligned
* Must be low-risk and easy to review

Examples:

* docs-canvas-agent-maint
* repo-sync-monitor-agent-monitor

---

### Review / Audit Agents

* Analyze and classify content
* Do not modify or delete content on first pass
* Produce structured audit reports

Examples:

* repo-recycle-audit-agent-review

---

# Repository Content Model

This repository contains multiple types of content. Agents must respect these boundaries.

---

## Generated Evidence (System-Owned)

Examples:

* artifacts/hc/
* logs/sessions/
* hw/
* timestamped healthcheck outputs
* systemd snapshots
* raw HC outputs

Characteristics:

* generated by automation
* append-only or snapshot-based
* not manually edited

Agent behavior:

* DO NOT rewrite or “improve” these
* may index, catalog, or summarize them only

---

## Curated Documentation (Human-Facing)

Examples:

* docs/
* observability-service-map.md
* telemetry-pipeline.md
* policy-observability.md
* index.md
* artifacts-index.md

Characteristics:

* derived from generated evidence
* intended for human understanding
* may become stale if not maintained

Agent behavior:

* may update navigation, structure, summaries
* should link to evidence rather than duplicate it
* should keep content scannable and accurate

---

## Control Plane (Repo Behavior)

Examples:

* README.md
* AGENTS.md
* docs/repo-content-model.md
* standards and navigation files

Characteristics:

* defines how the repo works
* consumed by Jules and contributors

Agent behavior:

* must be modified carefully
* changes should be small, intentional, and well explained

---

## Recycle / Quarantine

Examples:

* .recycle/

Characteristics:

* contains out-of-sync, duplicated, or uncertain material
* not authoritative

Agent behavior:

* DO NOT delete content on first pass
* classify and report only
* use audit agents to determine disposition

---

# Agent Operating Rules

## Scheduled Run Policy

Agents must prefer **no-op over noisy changes**.

Create a pull request only when:

* the change is small and self-contained
* the benefit is clear
* the change is low risk
* the review effort is minimal
* the change will not conflict with active local work

Otherwise:

* produce a report or do nothing

---

## Conflict Avoidance

Agents should avoid modifying:

* actively changing core files
* large documentation sections
* complex automation logic

Agents should prefer:

* index pages
* navigation blocks
* report artifacts
* clearly agent-owned files

---

## One-Run Discipline

Each agent run should:

* have one clear purpose
* produce at most one small PR
* or produce one report

---

## Output Standard

All agent outputs must include:

* Agent name
* Domain
* Role
* Mode
* What changed or what was found
* Why it matters
* Risk or watch items

---

# Report Locations

* docs/reports/hand-brief/

* docs/reports/delivery-scrum/

```text
latest.md
```

# Jules Integration

Jules automatically reads this file to understand:

* repository structure
* agent behavior
* where outputs should go
* how to safely operate

* use clear, scoped prompts
* reference agent definitions under `.github/agents/`
* prefer bounded, repeatable work
* avoid open-ended “improve everything” tasks
